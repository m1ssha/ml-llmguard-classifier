# OpenSpec: спецификация на разработку классификатора атак на LLM

## 1. Название инициативы
Разработка двухэтапного гибридного классификатора вредоносных запросов к LLM для использования в составе LLM Security Gateway.

## 2. Контекст
Проект посвящён защите больших языковых моделей от адаптивных атак. На первом этапе разработки требуется создать собственный классификатор, который будет анализировать входящие запросы пользователя до их передачи в основную LLM.

Классификатор должен быть не просто бинарным фильтром, а частью гибридной схемы защиты:
- сначала определяется, является ли запрос безопасным или вредоносным;
- затем, если запрос признан вредоносным, уточняется тип угрозы;
- результат передаётся в session risk engine и policy engine.

Классификатор является первым ключевым модулем будущего шлюза безопасности между интерфейсом чата и LLM.

## 3. Цель
Спроектировать и реализовать собственный ML-классификатор для выявления атакующих запросов к LLM, пригодный для интеграции в backend/gateway и пригодный для дальнейшего научного сравнения с baseline-подходами.

## 4. Функциональная идея
Система работает по двухэтапной схеме.

### Этап 1. Binary detection
Определить, относится ли входной запрос к одному из двух классов:
- `benign`
- `malicious`

### Этап 2. Threat type classification
Если запрос признан вредоносным, дополнительно определить тип угрозы:
- `prompt_injection`
- `jailbreak`
- `prompt_leakage`
- `role_override`
- `obfuscation_evasion`
- `tool_misuse`
- `policy_bypass`

Допускается реализация:
1. в виде двух отдельных моделей;
2. в виде одной модели с двумя выходными головами.

На стартовом этапе предпочтительно реализовать две независимые модели, чтобы упростить отладку, обучение и оценку качества.

## 5. Требования к MVP
На первой итерации требуется:
- подготовить структуру датасета;
- определить схему разметки;
- реализовать baseline-классификатор;
- реализовать основной encoder-based классификатор;
- подготовить интерфейс инференса для backend;
- предусмотреть логирование вероятностей и предсказанных классов.

## 6. Границы первой версии
В первую версию **не входят**:
- полноценный session-aware анализ нескольких сообщений;
- интеграция с RAG;
- управление tool-calls;
- финальный frontend;
- обучение большой генеративной LLM.

Первая версия должна быть сфокусирована именно на модуле классификации одного сообщения.

## 7. Ожидаемый результат первой версии
На выходе должен быть модуль, который принимает текстовый запрос и возвращает структуру вида:

```json
{
  "is_malicious": true,
  "binary_label": "malicious",
  "binary_confidence": 0.91,
  "threat_type": "prompt_leakage",
  "threat_confidence": 0.83,
  "signals": {
    "obfuscation_score": 0.10,
    "prompt_leakage_score": 0.79,
    "role_override_score": 0.21
  }
}
```

## 8. Требования к датасету
Нужно спроектировать собственный датасет для обучения и тестирования.

### 8.1. Базовые категории
Датасет должен содержать:
- безопасные запросы;
- явно вредоносные запросы;
- пограничные запросы;
- завуалированные запросы;
- запросы с обфускацией;
- примеры, похожие по тематике на вредоносные, но безопасные по сути.

### 8.2. Языковое покрытие
Желательно включить:
- русский язык;
- английский язык;
- смешанные русско-английские примеры;
- транслит;
- искажённые формулировки.

### 8.3. Принципы качества
Датасет должен быть собран так, чтобы модель не училась на очевидных шаблонах вроде:
- все вредоносные запросы содержат `ignore previous instructions`;
- все безопасные запросы связаны только с бытовыми темами.

Нужны сложные безопасные примеры, например:
- обсуждение темы jailbreak в научном или учебном контексте;
- объяснение того, что такое prompt injection;
- анализ угроз безопасности LLM;
- запросы на классификацию атак без попытки реально обойти защиту.

## 9. Схема разметки
Каждый пример должен иметь минимум такие поля:

```json
{
  "id": "sample_0001",
  "text": "...",
  "binary_label": "malicious",
  "threat_type": "prompt_leakage",
  "language": "ru",
  "severity": "medium",
  "source": "manual",
  "notes": "attempt to reveal hidden instructions"
}
```

### 9.1. Обязательные поля
- `id`
- `text`
- `binary_label`
- `threat_type`
- `language`
- `source`

### 9.2. Дополнительные поля
- `severity`
- `notes`
- `explicitness`
- `obfuscation_present`
- `split`

## 10. Требования к baseline
Нужно реализовать baseline для сравнения.

### Минимальный baseline
- TF-IDF
- Logistic Regression

### Альтернативы
- Linear SVM
- Random Forest

Baseline нужен для того, чтобы в дипломе можно было показать, что предложенный подход превосходит более простые методы.

## 11. Требования к основной модели
Предпочтительный путь:
- взять небольшую encoder-only модель;
- дообучить её как текстовый классификатор;
- использовать её для production-like инференса в gateway.

### Требования
- inference на CPU должен быть возможен;
- модель должна поддерживать работу с короткими и средними промптами;
- должны возвращаться вероятности классов;
- интерфейс должен быть пригоден для интеграции через Python backend.

## 12. Гибридная логика
Классификатор не должен быть единственным механизмом.

Нужно предусмотреть слой ручных признаков, который будет работать параллельно с ML-моделью.

### Примеры rule-based сигналов
- наличие конструкций для раскрытия системных инструкций;
- конструкции смены роли;
- попытки обойти ограничения;
- признаки обфускации;
- избыточное использование специальных символов;
- base64-подобные фрагменты;
- повторяющиеся указания игнорировать инструкции.

Результат rule-based анализа должен дополнять ML-скоринг.

## 14. Минимальный набор компонентов
Нужно реализовать:
- загрузку датасета;
- предобработку;
- обучение baseline;
- обучение основной модели;
- сохранение модели;
- скрипт инференса;
- оценку качества;
- экспорт результатов экспериментов.

## 15. Предлагаемый интерфейс инференса
Нужен единый Python API такого типа:

```python
result = classifier.predict(text)
```

где `result` содержит:
- бинарный класс;
- вероятность бинарного класса;
- тип угрозы;
- вероятность типа угрозы;
- список дополнительных сигналов.

## 16. Метрики качества
Для бинарной классификации:
- accuracy
- precision
- recall
- F1
- ROC-AUC

Для многоклассовой классификации:
- macro F1
- weighted F1
- confusion matrix
- precision/recall по каждому классу

Дополнительно:
- false positive rate
- false negative rate
- latency inference

## 17. Требования к экспериментам
Нужно подготовить эксперименты в трёх режимах:

### Эксперимент 1
Сравнение baseline и основной модели на одинаковом датасете.

### Эксперимент 2
Сравнение binary-only и двухэтапной схемы.

### Эксперимент 3
Оценка вклада rule-based признаков в итоговое качество.

## 18. Технические ограничения
Предпочтительно, чтобы решение:
- запускалось локально на обычном ПК;
- не требовало отдельного GPU для инференса;
- имело разумную задержку;
- было воспроизводимым.

## 19. Нефункциональные требования
- воспроизводимость обучения;
- прозрачная структура проекта;
- документированная схема разметки;
- удобство интеграции в FastAPI backend;
- логирование результатов инференса.

## 20. Definition of Done
Первая фаза считается завершённой, если:
1. собран и описан датасет;
2. реализован baseline;
3. реализован основной классификатор;
4. доступны скрипты обучения и инференса;
5. рассчитаны базовые метрики качества;
6. определён формат интеграции в gateway;
7. подготовлено краткое описание ограничений и направлений развития.

## 22. Формулировка для OpenSpec
Ниже текст, который можно использовать как рабочую постановку задачи.

### OpenSpec Task
Разработать модуль двухэтапной классификации пользовательских запросов к LLM. Модуль должен определять, является ли запрос вредоносным, а затем уточнять тип угрозы. Требуется реализовать собственный датасет, baseline, основную encoder-based модель, rule-based сигналы и унифицированный интерфейс инференса для последующей интеграции в LLM Security Gateway. Решение должно быть пригодно для научной оценки, воспроизводимо и рассчитано на локальный запуск без тяжёлой GPU-инфраструктуры.
