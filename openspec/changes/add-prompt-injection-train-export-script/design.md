## Context

Проект уже имеет модульную структуру для датасета, обучения, моделей, оценки и инференса, но production-поток остаётся фрагментированным между несколькими скриптами и внутренними вызовами. Для prompt injection сценария нужен один устойчивый entrypoint, который формирует inference-ready bundle без ручной сборки промежуточных шагов.

Ограничения:
- Переиспользовать существующие модули из `classifier/src/dataset/`, `classifier/src/training/`, `classifier/src/models/`, `classifier/src/evaluation/`.
- Сохранить совместимость с текущим inference-контрактом и форматом артефактов проекта.
- Обеспечить воспроизводимость через seed и явный manifest.

## Goals / Non-Goals

**Goals:**
- Реализовать один CLI-скрипт `classifier/scripts/run_prompt_injection_train_export.py` для end-to-end обучения и экспорта.
- Поддержать явные CLI-параметры: `dataset_id`, `output_root`, `seed`, `experiment_name`, `include_baseline`, `profile`.
- Нормализовать HF-датасет (`neuralchemy/Prompt-injection-dataset`) через alias-логику колонок/лейблов с понятными ошибками.
- Сформировать полный inference artifact bundle и единый `evaluation_report.json`.
- Добавить тесты (smoke + структура артефактов) и краткую документацию запуска.

**Non-Goals:**
- Переписывание существующих training pipeline или inference predictor с нуля.
- Добавление новых типов моделей, не используемых текущим encoder pipeline.
- Построение отдельного orchestration framework (Airflow/Prefect и т.п.).

## Decisions

1. **Оркестрация через единый thin-wrapper скрипт**
   - Решение: скрипт координирует шаги, а вычислительная/доменная логика остаётся в `classifier/src/*`.
   - Почему: минимизирует дублирование и риск расхождения поведения между train/eval/inference.
   - Альтернативы:
     - Дублировать логику в новом скрипте (отклонено: высокий риск drift).
     - Добавить магистральный класс-оркестратор в src (частично допустимо позже, но не обязательно для текущей цели).

2. **Нормализация входной схемы через alias map + fail-fast валидацию**
   - Решение: на этапе загрузки датасета скрипт использует заранее определённые алиасы для текстовой колонки и метки; при невозможности маппинга бросается понятная ошибка с перечислением ожидаемых/найденных полей.
   - Почему: HF-датасеты часто имеют вариации имён колонок; fail-fast ускоряет диагностику.
   - Альтернатива: молчаливые fallback-эвристики (отклонено: повышают риск скрытой деградации качества).

3. **Единый bundle-контракт для inference**
   - Решение: всегда экспортировать фиксированный набор файлов:
     - `encoder_model.joblib`
     - `tokenizer.joblib`
     - `label_mapping.json`
     - `thresholds.json`
     - `training_metadata.json`
     - `artifact_manifest.json`
     - `evaluation_report.json`
   - Почему: inference-слой должен загружать артефакты без доступа к training internals.
   - Альтернатива: условный состав bundle по профилю (отклонено: усложняет интеграцию и тестирование).

4. **Профилирование порогов и отчётности внутри одного запуска**
   - Решение: в `evaluation_report.json` фиксировать бинарные метрики минимум для `balanced` и `high_recall_security` независимо от выбранного runtime-профиля; выбранный `profile` влияет на active thresholds в `thresholds.json`.
   - Почему: сохраняется сравнимость качества между профилями и прозрачность для секьюрити-команды.
   - Альтернатива: считать только выбранный профиль (отклонено: теряется обязательная сравнительная информация).

5. **Опциональный baseline как дополнительная ветка**
   - Решение: при `include_baseline=true` запускать baseline pipeline через существующий модуль моделей/обучения, включать его метрики в отчёт отдельным блоком.
   - Почему: позволяет быстрый регрессионный бенчмарк без обязательного удлинения всех запусков.
   - Альтернатива: baseline всегда включён (отклонено: лишняя стоимость времени/ресурсов).

6. **Структура output по эксперименту**
   - Решение: сохранять результаты в `output_root/<experiment_name_or_timestamp>/` и писать относительные/абсолютные пути в manifest.
   - Почему: предотвращает перезапись и упрощает CI-архивацию.
   - Альтернатива: писать всегда в один каталог (отклонено: риск конфликтов и трудно воспроизводимые прогоны).

## Risks / Trade-offs

- **[Risk] Изменения схемы HF-датасета (split names, типы лейблов, колонок)** → **Mitigation:** строгая alias-валидация, детальные ошибки, unit/smoke тесты с мок-датасетом.
- **[Risk] Неполная совместимость экспортированных файлов с inference loader** → **Mitigation:** структура manifest и имена файлов фиксируются спецификацией, добавляется тест проверки bundle contract.
- **[Risk] Рост времени выполнения при include_baseline=true** → **Mitigation:** baseline по флагу, метрики baseline отделены и не блокируют основной экспорт encoder bundle.
- **[Risk] Разночтение профилей thresholding между обучением и продом** → **Mitigation:** явная фиксация активного профиля в `training_metadata.json` и полная таблица профилей в `thresholds.json`.

## Migration Plan

1. Добавить скрипт оркестрации и переиспользовать существующие src-модули.
2. Добавить слой нормализации HF-колонок/лейблов с alias-логикой и fail-fast ошибками.
3. Встроить экспорт bundle + manifest + evaluation report.
4. Добавить тесты и обновить `README.md`.
5. Прогнать smoke/pytest, зафиксировать артефакты в expected структуре.

Rollback стратегия:
- Изменения изолированы новым скриптом и тестами; при необходимости откат ограничивается удалением нового entrypoint и связанных тестов/документации без ломки текущих pipeline.
