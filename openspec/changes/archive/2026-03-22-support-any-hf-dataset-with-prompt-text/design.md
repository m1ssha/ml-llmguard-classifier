## Context

Текущий датасетный слой проекта классификатора атак на LLM должен поддерживать быстрый переход между источниками данных для экспериментов и обучения. На практике используется Hugging Face, но схемы разных датасетов отличаются: текст может быть в `text`, `prompt`, `user_input`, а целевая метка — в `label` с форматами `0/1`, `true/false`, bool/int/str. Также alias датасета часто «размазан» по коду, что повышает стоимость замены датасета и риск рассинхронизации.

Ограничения:
- Не ломать существующий контракт downstream-компонентов без необходимости.
- Сохранить совместимость с текущим стеком `datasets`.
- Обеспечить понятные ошибки валидации схемы на этапе загрузки, а не в середине пайплайна.

## Goals / Non-Goals

**Goals:**
- Сделать загрузчик универсальным для любого Hugging Face dataset alias.
- Явно закрепить правило обязательности текстового входа как `prompt` **или** `text` с поддержкой алиасов (например, `user_input`).
- Вынести alias в одно место в начале файла загрузчика для быстрой замены.
- Стандартизировать поведение ошибок при отсутствии обязательных полей.
- Добавить нормализацию `label` в бинарный формат (0/1) из `0/1`, `true/false`, bool/int/str.

**Non-Goals:**
- Редизайн полного training pipeline.
- Введение новой системы конфигурации на уровне всего проекта.
- Изменение бизнес-логики инференса и rule-based модуля.

## Decisions

1. Единый alias-константный параметр в начале загрузчика
- Решение: объявить значение HF alias один раз в верхней части `loader.py` (например, `HF_DATASET_ALIAS = "neuralchemy/Prompt-injection-dataset"`) и использовать его по умолчанию в функции загрузки.
- Почему: минимальная точка изменения, прозрачность для разработчика, отсутствие необходимости искать строковый литерал по проекту.
- Альтернатива: передавать alias только через CLI/ENV.
  - Причина отказа: в текущем объёме это избыточно; требование пользователя — быстро заменяемый alias в начале файла.

2. Универсальная загрузка через `datasets.load_dataset(alias, ...)`
- Решение: не хардкодить конкретные split/наборы колонок для одного источника; поддерживать любой alias с валидацией после загрузки.
- Почему: соответствует требованию «абсолютно любой датасет с Hugging Face».
- Альтернатива: allowlist поддерживаемых датасетов.
  - Причина отказа: противоречит требованию универсальности.

3. Вариативные входные колонки с правилом `prompt` ИЛИ `text`
- Решение: обновить схему/валидатор так, чтобы обязательным было наличие хотя бы одного канонического текстового поля (`prompt` или `text`), при этом поддерживать алиасы (`user_input` → `prompt`, при необходимости другие через конфиг/словарь маппинга).
- Почему: соответствует реальным схемам HF-датасетов и снижает количество ложных отказов загрузки.
- Альтернатива: требовать строго `prompt` и `text` одновременно.
  - Причина отказа: избыточно жёсткое требование, несовместимое с распространёнными датасетами.

4. Нормализация метки `label` в бинарный формат
- Решение: добавить слой нормализации `label` к `0/1` с поддержкой входов `0/1`, `true/false`, `True/False`, bool/int/str.
- Почему: унификация обучения/оценки, устранение неоднозначностей типов.
- Альтернатива: принимать только integer 0/1.
  - Причина отказа: теряется совместимость с большим числом публичных датасетов.

5. Минимальные правки downstream
- Решение: сохранить формат выходного DataFrame/датасета, добавив только гарантии присутствия `prompt` и `text`.
- Почему: снижает риск регрессий в обучении и метриках.

## Risks / Trade-offs

- [Риск] Некоторые HF-датасеты содержат нестандартные названия текстовых полей → [Смягчение] словарь алиасов и детальная ошибка с найденными колонками.
- [Риск] Универсальная загрузка может требовать настройки split (`train`, `test`) для конкретного датасета → [Смягчение] поддержать параметр split с дефолтом и понятной ошибкой при недоступном split.
- [Риск] Возможны неоднозначности при нормализации `label` (например, неожиданные строковые значения) → [Смягчение] строгий whitelist значений и fail-fast ошибка.
- [Trade-off] Правило `prompt` ИЛИ `text` повышает совместимость, но требует явной логики канонизации колонок.
- [Trade-off] Alias в коде проще менять вручную, но менее гибок, чем централизованный конфиг.

## Migration Plan

1. Обновить `loader.py`: добавить единый alias в начале файла и универсальную загрузку.
2. Обновить `schema.py`/`validator.py`: правило обязательности `prompt` ИЛИ `text` и поддержку алиасов (`user_input`).
3. Добавить нормализацию `label` до бинарного формата и ошибки для неподдерживаемых значений.
4. Обновить тесты датасетного слоя под новые требования.
5. Прогнать smoke/eval сценарии, убедиться в отсутствии регрессий.
6. Rollback: вернуть предыдущую версию загрузчика и схемы из VCS, если критические пайплайны перестанут проходить.
