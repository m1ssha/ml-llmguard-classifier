## Context

Инициатива задаёт первую фазу построения модуля защиты LLM, где классификатор работает до основной модели и возвращает структурированный риск-сигнал для gateway. Текущее состояние: требования описаны концептуально в `openspec_classifier_spec.md`, но не разложены в форму, удобную для пошаговой реализации, тестирования и интеграции.

Ограничения:
- MVP сфокусирован на анализе одного сообщения, без session-aware логики и RAG.
- Инференс должен быть выполним на CPU и пригоден для backend-интеграции.
- Нужна воспроизводимость экспериментов и корректное сравнение baseline vs encoder-based подхода.

Стейкхолдеры: ML-разработчик, backend/gateway разработчик, исследователь (оценка качества и сравнение подходов).

## Goals / Non-Goals

**Goals:**
- Спроектировать архитектуру MVP-классификатора из двух этапов: binary detection + threat type classification.
- Стандартизовать контракт данных: схема датасета, формат разметки, выходной JSON инференса.
- Определить поток обучения и оценки для baseline и основной модели с обязательными метриками.
- Определить место rule-based сигналов и способ их объединения с ML-скорингом.
- Обеспечить практическую интеграцию в Python backend/gateway через единый API предсказания.

**Non-Goals:**
- Реализация продвинутой многосообщенческой аналитики (session-aware).
- Интеграция с RAG/tool-calls и создание полноценного frontend.
- Обучение тяжёлой генеративной LLM или GPU-only решения.

## Decisions

1. **Двухэтапная архитектура как базовый путь MVP**
   - Решение: использовать отдельный бинарный классификатор и отдельный мультиклассовый классификатор угроз.
   - Почему: упрощает отладку, диагностику ошибок и поэтапный rollout.
   - Альтернатива: единая multi-head модель; откладывается как последующая оптимизация после стабилизации пайплайна.

2. **Разделение на capability-уровень: dataset/labeling, training/evaluation, inference interface**
   - Решение: оформить требования тремя отдельными capability в specs.
   - Почему: снижает связность, даёт явные acceptance-критерии и независимое тестирование.
   - Альтернатива: единый большой spec; отвергнуто из-за высокой сложности сопровождения.

3. **Гибридный скоринг (ML + rule-based сигналы)**
   - Решение: rule-based слой формирует дополнительные сигналы, которые возвращаются вместе с модельными вероятностями.
   - Почему: улучшает интерпретируемость и покрытие атак с обфускацией.
   - Альтернатива: только ML-предсказание; риск пропуска edge-case сценариев.

4. **Явный контракт инференса для gateway**
   - Решение: единый `predict(text)` контракт, который возвращает бинарный лейбл, confidence, threat type, threat confidence, signals.
   - Почему: минимизирует интеграционный риск и стабилизирует взаимодействие с downstream policy/risk engines.
   - Альтернатива: раздельные API для этапов; увеличивает сложность оркестрации на стороне backend.

5. **Экспериментальный протокол с baseline и абляциями**
   - Решение: обязательные 3 эксперимента — baseline vs encoder, binary-only vs two-stage, вклад rule-based слоя.
   - Почему: обеспечивает научно валидное сравнение и прозрачное обоснование выбранной архитектуры.
   - Альтернатива: единичная оценка на одной модели; недостаточно для принятия архитектурных решений.

## Risks / Trade-offs

- **[Risk]** Смещение датасета к очевидным паттернам (shortcut learning) → **Mitigation**: добавить сложные benign-примеры и пограничные случаи, ввести quality-гайдлайны разметки.
- **[Risk]** Высокий false positive rate на обсуждениях безопасности → **Mitigation**: отдельный фокус на precision/recall по классам и калибровка порогов.
- **[Risk]** Рост latency из-за двухэтапной схемы и rule-based анализа → **Mitigation**: CPU-профилирование, упрощение признаков, кеширование и оптимизация preprocessing.
- **[Risk]** Дублирование логики между training и inference preprocessing → **Mitigation**: единый модуль предобработки и инвариантные тесты соответствия.
- **[Trade-off]** Раздельные модели проще поддерживать, но сложнее оптимизировать end-to-end качество.
- **[Trade-off]** Богатый JSON-ответ улучшает объяснимость, но увеличивает интеграционный контракт и требования к версионированию.
